\title{AgentProf: Semantic Profiling for AI Agents}

\begin{abstract}
%% S1: Context (→¶1)
AI agents increasingly orchestrate long-running, multi-step
activities involving thousands to millions of interactions with
users, tools, and system resources.
% AI 智能体编排长时间运行的多步骤活动，涉及与用户、工具和系统资源的数千到数百万次交互。
%% S2: Need (→¶2)
To improve quality, safety, and cost efficiency, developers need to
determine which task categories consume the most budget, where
failures concentrate, and which workflows trigger unsafe effects.
% 为了提高质量、安全性和成本效率，开发者需要确定哪类任务消耗最多预算、故障集中在哪里、哪些工作流触发不安全副作用。
%% S3: Traditional profiling (→¶2+¶4)
To answer similar questions in traditional systems software, profiling
attributes resource consumption to responsible entities, such as code
paths, by aggregation, rather than single-execution debugging or
tracing.
% 在传统系统软件中，profiling 通过聚合将资源消耗归因到负责实体（如代码路径），而非单次执行调试或 tracing。
%% S4: Gap (→¶2+¶3)
Yet existing agent observability tools are mainly for debugging and
tracing instead of profiling.
% 然而现有智能体可观测性工具主要用于调试和 tracing，而非 profiling。
%% S5: Agent challenge (→¶4)
Agent behavior differs fundamentally from code execution.
The responsible entities are semantic categories such as task intent
and workflow phase, not code paths, and agent trajectories lack the
deterministic identifiers and structural hierarchies that make
aggregation possible.
% 智能体行为与代码执行有根本不同。负责实体是语义类别（如任务意图和工作流阶段），而非代码路径，且智能体轨迹缺乏使聚合成为可能的确定性标识符和结构层次。
%% S6: Model (→¶5)
We propose a \emph{semantic operation stack model}.
Uniform \emph{operations} represent all agent activities, and
\emph{operation stacks} replace the runtime call stack with
query-time field projections that fold identical frame sequences.
% 我们提出语义 operation stack 模型。统一的 operation 表示所有智能体活动，operation stack 用查询时字段投影替代运行时调用栈，折叠相同帧序列。
%% S7: System (→¶6)
\sys implements this model as a profiler with pluggable algorithms
for \emph{intent attribution} and \emph{stack construction},
compiling local agent trajectories into pprof-compatible profiles.
% \sys 将此模型实现为具有可插拔意图归因和栈构造算法的 profiler，将本地智能体轨迹编译为 pprof 兼容的 profile。
%% S8-S9: Results (→¶6)
On 325 real agent sessions, semantic profiling separates over 90\% of
system-effect weight that remains mixed under session-level views.
% 在 325 个真实智能体会话上，语义 profiling 分离了会话级视图所混合的超过 90\% 的系统副作用权重。
On four public datasets with hidden ground-truth labels
(34{,}539 operations), profiler-ranked groups localize labeled
problems at 9.4\% inspection cost versus flat summaries, with 45\%
fewer groups than fixed-session drilldown.
% 在四个具有隐藏真实标签的公开数据集（34,539 个操作）上，profiler 排序的分组以 9.4\% 的检查工作量定位标注问题，比固定会话下钻减少 45\% 的分组数。
\end{abstract}

\section{Introduction}
\label{sec:intro}

%% ¶1: Background + context
AI agents orchestrate multi-step activities that span two
layers, high-level intent (prompts, LLM calls, tool invocations) and
low-level system effects (process spawns, file and network I/O).
% AI 智能体编排跨越两层的多步骤活动：高层意图（提示词、LLM 调用、工具调用）和低层系统副作用（进程创建、文件和网络 I/O）。
As agents evolve from short, scripted workflows into complex systems
that run for hours and days, with a large number of interactions each, production
teams accumulate many such trajectories over weeks or months.
% 随着智能体从短小的脚本化工作流进化为每次运行数小时、包含数千次交互的复杂系统，生产团队在数周或数月内积累大量此类轨迹。

%% ¶2: Problem
To improve agent quality, safety, and cost efficiency, developers
need to analyze operational behavior across trajectories and
interactions.
Which categories of tasks consume the most token budget? Where do
failures concentrate across workflows? Which behavioral patterns
trigger unsafe system effects?
% 为了提高智能体质量、安全性和成本效率，开发者需要跨轨迹和交互分析运维行为。哪类任务消耗最多 token 预算？故障跨工作流集中在哪里？哪些行为模式触发不安全系统副作用？
Answering these questions requires analysis and evaluation across many
trajectories, a task becoming costly for manual inspection or
per-trajectory LLM evaluation~\cite{llm-as-judge} at scale.
% 回答这些问题需要跨许多轨迹的分析和评估，在规模化时人工检查或逐轨迹 LLM 评估的成本越来越高。
In traditional systems software, profiling answers these questions by
attributing resource consumption to responsible entities by
aggregation, as distinct from single-execution debugging and tracing.
% 在传统系统软件中，profiling 通过聚合将资源消耗归因到负责实体来回答这些问题，区别于单次执行调试和 tracing。

%% ¶3: Existing solutions
Yet existing agent tools support debugging and tracing but not
profiling.
% 然而现有智能体工具支持调试和 tracing，但不支持 profiling。
LangSmith~\cite{langsmith}, Langfuse~\cite{langfuse},
Phoenix~\cite{phoenix}, and OpenTelemetry GenAI~\cite{otel} organize
events into per-session span trees, effective for diagnosing a single
run but unable to aggregate by semantic category.
% LangSmith~\cite{langsmith}、Langfuse~\cite{langfuse}、Phoenix~\cite{phoenix} 和 OpenTelemetry GenAI~\cite{otel} 将事件组织为按会话的 span 树，有效诊断单次运行但无法按语义类别跨会话聚合。
Datadog LLM Observability~\cite{datadog-llmobs} clusters user
messages by topic, and Laminar~\cite{laminar-signals} extracts
structured signals from traces, but both characterize \emph{input
distributions} (``30\% of users ask code questions'') rather than
\emph{resource attribution} (``review tasks consume 40\% of the
token budget''), because labels at the request level do not propagate
to downstream tool calls and system effects.
% Datadog LLM Observability~\cite{datadog-llmobs} 按主题聚类用户消息，Laminar~\cite{laminar-signals} 从 trace 中提取结构化信号，但两者都刻画输入分布（"30\% 的用户提问代码问题"）而非资源归因（"review 任务消耗了 40\% 的 token 预算"），因为请求级的标签不会传播到下游工具调用和系统副作用。

%% ¶4: Challenge — why traditional profiling doesn't transfer directly
Adapting profiling to agent trajectories is challenging
(\S\ref{sec:background}).
% 将 profiling 适配到智能体轨迹具有挑战性（\S\ref{sec:background}）。
In traditional software, the responsible entities are code paths.
Profiling is straightforward because function names are
deterministic and runtime call nesting provides the aggregation
hierarchy, and flame graphs~\cite{flamegraphs} and pprof~\cite{pprof} call graphs visualize the result.
% 在传统软件中，负责实体是代码路径。Profiling 是直接的，因为函数名是确定性的，运行时调用嵌套提供聚合层次，火焰图~\cite{flamegraphs}可视化其结果。
Agent behavior differs fundamentally.
To answer the questions above, the responsible entities need to be
semantic categories such as task intent and tool operation, not code
paths.
% 智能体行为有根本不同。要回答上述问题，负责实体需要是语义类别（如任务意图和工具操作），而非代码路径。
Agent trajectories also lack the two properties that make aggregation possible.
% 智能体轨迹还缺乏使聚合成为可能的两个特性。
Prompts are natural language, and two prompts expressing the same intent share no common string.
% 提示词是自然语言，表达相同意图的两个提示词没有共同字符串。
Framework-level span names (\texttt{LLMCall}, \texttt{ToolInvoke}) describe mechanisms, not semantic intent.
% 框架级 span 名称（\texttt{LLMCall}、\texttt{ToolInvoke}）描述机制，而非语义意图。
Agent events have no runtime call-stack hierarchy because prompts,
tool calls, and system effects are not nested by execution the way
function calls produce stack frames.
% 智能体事件没有运行时调用栈层次，因为提示词、工具调用和系统副作用不像函数调用那样通过执行嵌套产生栈帧。

%% ¶5: Model
Despite these differences, the fundamental profiling
methodology (define units, assign stable labels, construct
aggregation hierarchies, fold identical stacks) transfers to agent
trajectories.
% 尽管有这些差异，基本的 profiling 方法论（定义单元、赋予稳定标签、构造聚合层次、折叠相同栈）可以迁移到智能体轨迹。
We propose a \emph{semantic operation stack model} with two
components.
% 我们提出一个包含两个组件的语义 operation stack 模型。
\emph{Operations} are uniform records with string fields and
additive measures that represent all agent activities (prompts, LLM
calls, tool invocations, and system effects) without type-specific
objects.
% Operation 是带有字符串字段和可加度量的统一记录，表示所有智能体活动（提示词、LLM 调用、工具调用和系统副作用），无需特定类型对象。
\emph{Operation stacks} are ordered lists of fields chosen at query
time.
% Operation stack 是在查询时选择的字段有序列表。
Each operation is projected to a frame sequence, and operations
with identical sequences merge, producing the folded-stack
representation that flame graphs visualize
(Figure~\ref{fig:flamegraph}).
% 每个 operation 被投影到帧序列，具有相同序列的 operation 合并，产生火焰图可视化的折叠栈表示。
Different field selections answer different questions (by task
category, by execution phase, by session, or by action type) from
the same data.
% 不同的字段选择从同一数据回答不同问题（按任务类别、按执行阶段、按会话或按操作类型）。

%% ¶6: This paper + system + results
We implement this model in \sys, an offline profiler that compiles
local agent trajectories into pprof-compatible profiles
(Figure~\ref{fig:architecture}).
% 我们在 \sys 中实现此模型，这是一个离线 profiler，将本地智能体轨迹编译为 pprof 兼容的 profile（Figure~\ref{fig:architecture}）。
Two pluggable algorithm frameworks enrich the model.
% 两个可插拔算法框架丰富此模型。
\emph{Intent attribution} derives deterministic, low-cardinality labels from natural-language fields via regex rules, a local LLM tagger, or unsupervised clustering.
% 意图归因通过正则规则、本地 LLM 标注器或无监督聚类从自然语言字段派生确定性、低基数的标签。
This restores the folding precondition that CPU profiling obtains for free from function names.
% 这恢复了 CPU profiling 从函数名免费获得的折叠前提条件。
\emph{Stack construction} builds aggregation hierarchies from
available fields, either from a user-specified ordered list of fields or via
automatic induction that scores semantic shifts and partition
coherence.
% 栈构造从可用字段构建聚合层次，可通过用户指定的有序字段列表或通过评分语义转移和分区一致性的自动诱导。
On 325 real Codex and Claude sessions (183{,}714 system observations),
semantic profiling separates over 90\% of system-effect weight that
remains mixed under session-level views.
% 在 325 个真实 Codex 和 Claude 会话（183,714 个系统观测）上，语义 profiling 分离了会话级视图所混合的超过 90\% 的系统副作用权重。
On four public datasets with hidden ground-truth labels
(34{,}539 operations), profiler-ranked groups localize labeled
problems at 9.4\% inspection work versus flat summaries, with 45\%
fewer groups than fixed-session drilldown.
% 在四个具有隐藏真实标签的公开数据集（34,539 个操作）上，profiler 排序的分组以 9.4\% 的检查工作量定位标注问题，比固定会话下钻减少 45\% 的分组数。
Semantic-label quality exceeds V-measure 0.7 on 7 of 9 held-out datasets.
% 语义标签质量在 9 个保留数据集中的 7 个上超过 V-measure 0.7。

%% ¶7: Contributions
This paper makes three contributions:
% 本文做出三项贡献：

\begin{enumerate}
\item \textbf{Semantic operation stack model.}
  We identify the missing profiling layer in agent observability and
  propose a model of uniform operations and query-time operation
  stacks (\S\ref{sec:background}--\S\ref{sec:design}).
% 语义 operation stack 模型。识别智能体可观测性中缺失的 profiling 层，提出统一 operation 和查询时 operation stack 的模型。
\item \textbf{System: \sys.}
  A profiler that implements this model with pluggable intent
  attribution and stack construction, producing pprof-compatible
  output (\S\ref{sec:impl}).
% 系统：\sys。实现此模型的 profiler，具有可插拔意图归因和栈构造，产生 pprof 兼容输出。
\item \textbf{Hidden-label evaluation.}
  We evaluate profiler output against hidden ground-truth trajectory
  labels, analogous to injecting known bottlenecks in CPU profiler
  evaluation, on 325 real sessions and 4 public datasets
  (\S\ref{sec:eval}).
% 隐藏标签评估。将 profiler 输出与隐藏的真实轨迹标签评分，类似于在 CPU profiler 评估中注入已知瓶颈，在 325 个真实会话和 4 个公开数据集上验证。
\end{enumerate}

\section{Background and Motivation}
\label{sec:background}

\subsection{AI Agent Observability}
% AI 智能体可观测性

As introduced in \S\ref{sec:intro}, agent activities span both intent and system-effect layers.
% 如 \S\ref{sec:intro} 所述，智能体活动跨越意图层和系统副作用层。
Existing observability tools cover these layers separately.
% 现有可观测性工具分别覆盖这两层。
LangSmith~\cite{langsmith}, Langfuse~\cite{langfuse}, Phoenix~\cite{phoenix}, and OpenTelemetry GenAI~\cite{otel} organize agent-level events into per-execution \emph{span trees}.
% LangSmith~\cite{langsmith}、Langfuse~\cite{langfuse}、Phoenix~\cite{phoenix} 和 OpenTelemetry GenAI~\cite{otel} 将智能体级事件组织为按执行划分的 span 树。
System monitors show process and file activity without semantic attribution.
% 系统监控器显示进程和文件活动，但没有语义归因。
The span-tree design is effective for debugging.
% span 树设计对调试是有效的。
A developer can locate a failing span, inspect its attributes, and trace its parent chain.
% 开发者可以定位失败的 span、检查其属性并追踪其父链。
AgentSight~\cite{agentsight} bridges the two layers by correlating eBPF-captured system events with intercepted LLM traffic, but the resulting traces still lack aggregate profiling.
% AgentSight~\cite{agentsight} 通过关联 eBPF 捕获的系统事件与拦截的 LLM 流量来桥接两层，但生成的 trace 仍缺乏聚合分析能力。

\subsection{Profiling vs.\ Debugging}
% 性能分析 vs. 调试

Debugging inspects a single execution, but profiling aggregates across executions to answer questions like ``where does the budget go?''
% 调试检查单次执行，而性能分析跨执行聚合以回答"预算花在哪里？"等问题。
CPU profilers solved this aggregation problem decades ago.
% CPU 分析器几十年前就解决了这个聚合问题。
A flame graph~\cite{flamegraphs} represents a profile as weighted \emph{samples}, each attached to a \emph{call stack}.
% 火焰图~\cite{flamegraphs}将性能概要表示为加权样本，每个样本附带一个调用栈。
Identical stacks merge, and width represents aggregate cost.
% 相同的栈合并，宽度代表聚合开销。
The key enabler is \emph{deterministic} function names, so the same code path always produces the same frames.
% 关键条件是确定性的函数名，使相同代码路径始终产生相同的栈帧。

Agent activities break this precondition in two ways.
% 智能体活动从两方面打破了这一前提。
First, prompts are natural language and lack deterministic identifiers.
% 第一，提示词是自然语言，缺乏确定性标识符。
Second, agent events have no inherent call stack because they lack the runtime-determined parent-child nesting that function calls provide.
% 第二，智能体事件没有固有的调用栈，因为它们缺乏函数调用提供的运行时确定的父子嵌套关系。
Pprof's \texttt{tagroot}/\texttt{tagleaf} options add label-derived pseudo-frames to an existing execution stack, but agent activities have no execution stack to augment.
% Pprof 的 \texttt{tagroot}/\texttt{tagleaf} 选项在已有执行栈上添加标签衍生的伪帧，但智能体活动没有执行栈可供扩展。

\subsection{The Aggregation Gap}
% 聚合缺口

Existing agent observability tools are designed for debugging, not profiling, and this creates three concrete gaps.
% 现有智能体可观测性工具为调试而设计，而非性能分析，这造成了三个具体缺口。

First, sessions are boundaries, not categories.
% 第一，会话是边界，不是类别。
A single session may contain review, debugging, and refactoring phases, and splitting by session assigns all of them to the same bucket.
% 单个会话可能包含 review、debugging 和 refactoring 阶段，按会话拆分会将它们全部分配到同一桶中。
Two sessions performing the same task produce separate buckets that cannot be merged without external logic.
% 执行相同任务的两个会话会产生无法在没有外部逻辑的情况下合并的分离桶。

Second, span names describe mechanisms (\texttt{LLMCall}, \texttt{ToolInvoke}, \texttt{ChatCompletion}), not user intent (\texttt{review}, \texttt{debug}, \texttt{test}).
% 第二，span 名称描述机制（\texttt{LLMCall}、\texttt{ToolInvoke}、\texttt{ChatCompletion}），而非用户意图（\texttt{review}、\texttt{debug}、\texttt{test}）。
Grouping by span name tells you how many LLM calls occurred, not which categories of work consumed the budget.
% 按 span 名称分组只能告诉你发生了多少次 LLM 调用，而非哪些工作类别消耗了预算。

Third, these limitations compound in practice.
% 第三，这些限制在实践中叠加放大。
A common system effect like running a test suite may appear thousands of times across sessions, but a flat summary shows only a single counter with no way to attribute executions to the semantic intent that triggered them.
% 像运行测试套件这样的常见系统副作用可能在会话中出现数千次，但扁平摘要只显示一个计数器，无法将执行归因到触发它们的语义意图。
We quantify this in RQ1~(\S\ref{sec:eval}).
% 我们在 RQ1~(\S\ref{sec:eval}) 中量化了这一点。

These observations yield three design requirements:
% 这些观察得出三个设计需求：

\begin{description}
\item[R1: Uniform record type.]
Agent activities mix prompts, LLM calls, tool invocations, GUI actions, process events, and benchmark labels.
% 智能体活动混合了提示词、LLM 调用、工具调用、GUI 操作、进程事件和基准标签。
The profiler needs a single record type that can represent all of them without type-specific objects.
% 分析器需要一种能表示所有这些活动的单一记录类型，无需特定类型的对象。
\item[R2: Query-time aggregation.]
The same data must support multiple aggregation shapes (by task, phase, action, session, or dataset) without rebuilding the input.
% 同一数据必须支持多种聚合形状（按任务、阶段、操作、会话或数据集），无需重建输入。
Sessions and spans must be fields, not fixed hierarchy nodes.
% 会话和 span 必须是字段，而非固定层次节点。
\item[R3: Deterministic labels for folding.]
Free-form prompts must be mapped to short, stable, repeatable labels before folding.
% 自由形式的提示词必须在折叠前映射为短小、稳定、可重复的标签。
This \emph{intent attribution} step has no analogue in CPU profiling, where function names are already deterministic.
% 这一意图识别步骤在 CPU 性能分析中没有对应物，因为函数名已经是确定性的。
\end{description}

\section{Design}
% 设计
\label{sec:design}

The three requirements (a uniform record type, query-time aggregation, and deterministic labels for folding) drive the design.
% 三个需求（统一记录类型、查询时聚合和用于折叠的确定性标签）驱动了设计。
The semantic operation stack model has two components, operations (\S\ref{sec:ops}) and operation stacks (\S\ref{sec:stacks}), satisfying R1 and R2.
% 语义 operation stack 模型有两个组件——operation（\S\ref{sec:ops}）和 operation stack（\S\ref{sec:stacks}）——满足 R1 和 R2。
Two pluggable algorithm frameworks enrich the model. Intent attribution (\S\ref{sec:intent}) satisfies R3 by deriving deterministic labels, and stack construction (\S\ref{sec:stacks}) builds aggregation hierarchies from available fields.
% 两个可插拔算法框架丰富此模型。意图归因（\S\ref{sec:intent}）通过派生确定性标签满足 R3，栈构造（\S\ref{sec:stacks}）从可用字段构建聚合层次。
The profiling pipeline has five stages: (1)~parse raw traces into operations, (2)~enrich operations via intent attribution or mapping rules, (3)~filter by a user-specified predicate, (4)~project each operation onto a stack frame sequence chosen at query time, and (5)~fold operations with identical stacks, summing their weights.
% 分析管线有五个阶段：(1) 将原始 trace 解析为 operation，(2) 通过意图归因或映射规则丰富 operation，(3) 按用户指定的谓词过滤，(4) 将每个 operation 投影到查询时选择的栈帧序列上，(5) 折叠具有相同栈的 operation 并求和权重。

\subsection{Semantic Operation Stack Model}
% 语义 Operation Stack 模型
\label{sec:ops}

%% ¶1: Operations
Because agent activities span both the intent and system-effect
layers described in \S\ref{sec:background}, a profiler should not
distinguish between them at the object-model level.
% 由于智能体活动跨越 \S\ref{sec:background} 中描述的意图层和系统副作用层，分析器不应在对象模型层面区分它们。
\sys therefore represents every activity as a single abstraction, the
\operation, a weighted record with string fields and additive
measures.
% 因此 \sys 将每个活动表示为单一抽象——operation，一个带有字符串字段和可加度量的加权记录。
Each operation carries string fields (\texttt{project},
\texttt{agent}, \texttt{session}, \texttt{prompt\_tag},
\texttt{kind}, \texttt{model}, \texttt{path}, \texttt{domain},
\texttt{status}, \ldots) and additive measures (token count,
duration, occurrence count).
% 每个 operation 携带字符串字段（\texttt{project}、\texttt{agent}、\texttt{session}、\texttt{prompt\_tag}、\texttt{kind}、\texttt{model}、\texttt{path}、\texttt{domain}、\texttt{status} 等）和可加度量（token 数、持续时间、出现次数）。
A prompt, an LLM call, a tool invocation, a file read, a GUI click,
and a process event are all operation records that differ only in
which fields are populated.
% 提示词、LLM 调用、工具调用、文件读取、GUI 点击和进程事件都是 operation 记录，仅在填充的字段上有所不同。

%% ¶2: Operation Stacks
\label{sec:stacks}
An \opstack is an ordered list of fields chosen at query time.
% 一个 operation stack 是在查询时选择的字段有序列表。
Given a stack specification $[f_1, f_2, \ldots, f_k]$, each
operation $o$ is projected to a frame sequence
$\langle o.f_1, o.f_2, \ldots, o.f_k \rangle$.
% 给定栈规范 $[f_1, f_2, \ldots, f_k]$，每个 operation $o$ 被投影到帧序列 $\langle o.f_1, o.f_2, \ldots, o.f_k \rangle$。
Operations with identical frame sequences are merged and their
weights are summed.
% 具有相同帧序列的 operation 被合并，权重求和。
Unlike CPU profiling, where the call stack is determined by
execution, the stack is determined by the query.
% 与 CPU 分析中调用栈由执行决定不同，这里的栈由查询决定。
The same operations can be folded as a flat task summary, a
fixed-session tree, a semantic phase/action hierarchy, or a
dataset-native trajectory, and changing the fields changes the
aggregation without changing the data.
% 同一组 operation 可以折叠为扁平任务摘要、固定会话树、语义阶段/操作层次或数据集原生轨迹，改变字段即改变聚合方式而无需改变数据。
Sessions and spans are optional operation fields, not structural
skeleton nodes, so the same data supports both debugging views
(include \texttt{session}) and aggregate profiling views (exclude
it).
% 会话和 span 是可选的 operation 字段而非结构性骨架节点，因此同一数据既支持调试视图（包含 \texttt{session}）也支持聚合 profiling 视图（排除它）。
A view is a triple $(\varphi, \sigma, w)$: a predicate
$\varphi$ selects which operations participate, a stack function
$\sigma = [f_1, \ldots, f_k]$ maps each operation to its frame
sequence, and a weight function $w$ assigns a nonnegative measure.
% 一个视图是三元组 $(\varphi, \sigma, w)$：谓词 $\varphi$ 选择参与的 operation，栈函数 $\sigma = [f_1, \ldots, f_k]$ 将每个 operation 映射到其帧序列，权重函数 $w$ 赋予非负度量。
Four built-in defaults (Table~\ref{tab:views}) answer common
questions.
% 四个内置默认视图（Table~\ref{tab:views}）回答常见问题。

\subsection{Algorithms}
% 算法
\label{sec:intent}

\paragraph{Intent attribution.}
% 意图归因。
Intent attribution maps free-form prompts to short, stable,
deterministic labels, restoring the folding precondition that agent
prompts violate (\S\ref{sec:background}).
% 意图归因将自由形式的提示词映射为短小、稳定、确定性的标签，恢复智能体提示词所违反的折叠前提条件（\S\ref{sec:background}）。
The framework supports three pluggable backends.
% 该框架支持三种可插拔后端。
Regex rules are the production default: rules have the form
\texttt{KIND:TAG=REGEX}, are evaluated in order (first match wins),
and the developer iterates with \sys until the unmatched rate drops
below 5\%, typically in 5--10 rounds.
% 正则规则是生产环境默认：规则格式为 \texttt{KIND:TAG=REGEX}，按顺序求值（首个匹配胜出），开发者与 \sys 迭代直到未匹配率降至 5\% 以下，通常需要 5--10 轮。
A local LLM tagger (e.g., a quantized 3B-parameter model via llama.cpp)
generates a single-word label per prompt with grammar-constrained
decoding (forcing output to match a predefined token grammar) and caching.
% 本地 LLM 标注器（如通过 llama.cpp 运行的量化 3B 参数模型）通过语法约束解码（强制输出匹配预定义 token 语法）和缓存为每个提示词生成单词标签。
Unsupervised clustering via TF-IDF and K-Means discovers
natural prompt groups without predefined rules, feeding rule
authoring.
% 通过 TF-IDF 和 K-Means 的无监督聚类在没有预定义规则的情况下发现自然提示词分组，为规则编写提供输入。
For structured traces where action types and quality annotations are
explicit fields, mapping rules derive new fields from existing
ones (e.g., \texttt{phase:navigate=type:(click|scroll|key)}).
% 对于操作类型和质量标注是显式字段的结构化 trace，映射规则从现有字段派生新字段（如 \texttt{phase:navigate=type:(click|scroll|key)}）。
All backends are pluggable, and a profiling configuration records the
complete analysis setup for deterministic replay.
% 所有后端都是可插拔的，profiling configuration 记录完整的分析配置以实现确定性重放。

\paragraph{Stack construction.}
% 栈构造。
Stack construction builds aggregation hierarchies from
available fields.
% 栈构造从可用字段构建聚合层次。
When users supply a ordered list of fields, the projection is direct.
% 当用户提供有序字段列表时，投影是直接的。
When no chain is supplied, \sys can induce operation stacks
automatically by scoring adjacent operations for semantic shifts,
field changes, and partition coherence, then cutting at boundaries
where the evidence is strongest.
% 当不提供有序字段列表时，\sys 可以通过对相邻 operation 评分语义转移、字段变化和分区一致性来自动诱导 operation stack，然后在证据最强的边界处切分。
The induced stacks have variable depth, and a configurable depth cap
controls the granularity.
% 诱导的栈具有可变深度，可配置的深度上限控制粒度。
This mode produces coarser groups than hand-specified stacks but
requires no domain knowledge, making it useful for initial
exploration (\S\ref{sec:eval}).
% 此模式产生比手动指定栈更粗的分组，但不需要领域知识，适用于初始探索（\S\ref{sec:eval}）。

\section{Implementation}
% 实现
\label{sec:impl}

\sys is an offline, post-hoc profiler implemented as a Rust CLI (Figure~\ref{fig:architecture}).
% \sys 是一个离线、事后分析器，实现为 Rust CLI（Figure~\ref{fig:architecture}）。
It reads recorded traces, not live events.
% 它读取记录的 trace，而非实时事件。
The \texttt{agent-session} crate reads Codex and Claude Code local JSONL history files and reconstructs prompts, LLM calls, tool invocations, and their triggered file and network effects.
% \texttt{agent-session} crate 读取 Codex 和 Claude Code 本地 JSONL 历史文件，重建提示词、LLM 调用、工具调用及其触发的文件和网络副作用。
Each prompt opens a span.
% 每个提示词打开一个 span。
LLM calls, tool calls, and system effects within that span inherit its semantic label, connecting intent to system behavior without runtime instrumentation.
% 该 span 内的 LLM 调用、工具调用和系统副作用继承其语义标签，无需运行时插桩即可连接意图与系统行为。
Public benchmark datasets enter as operation JSONL with mapping rules deriving fields before stack construction.
% 公开基准数据集以 operation JSONL 形式进入，映射规则在栈构建之前派生字段。
Standard Chrome Trace Event format can also be imported, with non-\sys protocol fields preserved as operation fields.
% 标准 Chrome Trace Event 格式也可导入，非 \sys 协议字段作为 operation 字段保留。

Output formats include pprof protobuf (compatible with \texttt{go tool pprof -http}), folded-stack text (compatible with \texttt{flamegraph.pl}), standalone SVG flame graphs, and JSON summaries.
% 输出格式包括 pprof protobuf（兼容 \texttt{go tool pprof -http}）、折叠栈文本（兼容 \texttt{flamegraph.pl}）、独立 SVG 火焰图和 JSON 摘要。
Operations can also be exported as standard Chrome Trace Event JSON and re-imported with byte-identical folded output, enabling interoperability with trace viewers such as Perfetto.
% Operation 也可以导出为标准 Chrome Trace Event JSON 并重新导入，折叠输出字节相同，支持与 Perfetto 等 trace 查看器的互操作。
All outputs are deterministic given the same input and profiling configuration.
% 在相同输入和 profiling configuration 下，所有输出都是确定性的。

\sys is the offline profiling component of the AgentSight~\cite{agentsight} observability framework.
% \sys 是 AgentSight~\cite{agentsight} 可观测性框架的离线分析组件。
AgentSight provides real-time eBPF-based capture of SSL traffic, process events, and file/network effects.
% AgentSight 提供基于 eBPF 的实时 SSL 流量、进程事件和文件/网络副作用捕获。
\sys aggregates the recorded traces into semantic profiles, so the two are complementary: AgentSight answers ``what happened during this run'' (debugging), and \sys answers ``where did the budget go across runs'' (profiling).
% \sys 将记录的 trace 聚合为语义概要，因此两者互补：AgentSight 回答"这次运行发生了什么"（调试），\sys 回答"预算跨运行花在了哪里"（分析）。

\subsection{RQ1: Does Semantic Profiling Improve Resource Attribution?}
% RQ1：语义分析能否改善资源归因？

The strongest argument against semantic profiling is that traditional tools ``can already see the same information.''
% 反对语义分析的最强论点是传统工具"已经能看到相同信息"。
To test this, we run a semantic-axis ablation on 325 real Codex/Claude sessions (183{,}714 system observations).
% 为了检验这一点，我们在 325 个真实 Codex/Claude 会话（183,714 个系统观测）上进行语义轴消融。
We progressively add label axes (no labels, session only, prompt only, both) and measure the mixed-weight percentage, the fraction of system-effect weight in buckets that mix observations from multiple task categories.
% 我们逐步添加标签轴（无标签、仅会话、仅提示词、两者兼有），测量混合权重百分比——混合了多个任务类别观测的桶中的系统副作用权重比例。
A lower mixed-weight percentage means the profiler correctly separates effects by task category.
% 更低的混合权重百分比意味着分析器正确地按任务类别分离了效果。
The residual percentage measures weight in buckets where the majority category accounts for less than 90\% of the bucket, capturing partial mixing.
% 残余百分比测量多数类别占桶内不到 90\% 的权重，捕获部分混合。

\centering\small

\label{tab:mixing}
\begin{tabular}{lrrr}
\toprule

\midrule

\bottomrule

Table~\ref{tab:mixing} shows the result on 325 real Codex/Claude sessions.
% Table~\ref{tab:mixing} 展示了 325 个真实 Codex/Claude 会话上的结果。
Without semantic labels, 90.4\% of system-effect weight falls into mixed buckets.
% 没有语义标签时，90.4\% 的系统副作用权重落入混合桶。
A flat \texttt{cargo~test} counter shows 2{,}903 executions but cannot distinguish whether they came from \texttt{review}, \texttt{debug}, \texttt{refactor}, or \texttt{test} prompts.
% 扁平的 \texttt{cargo test} 计数器显示 2,903 次执行，但无法区分它们来自 \texttt{review}、\texttt{debug}、\texttt{refactor} 还是 \texttt{test} 提示词。
Adding only the prompt label reduces mixed weight to 36.7\% and residual to 7.5\%.
% 仅添加提示词标签就将混合权重降至 36.7\%，残余降至 7.5\%。
The session label alone barely helps (84.4\%), because sessions typically contain multiple intent phases.
% 仅会话标签几乎没有帮助（84.4\%），因为会话通常包含多个意图阶段。

A label-permutation test (1{,}000 shuffles, $p = 0.001$) confirms the separation is not random.
% 标签置换测试（1,000 次洗牌，$p = 0.001$）确认分离不是随机的。
Semantic-label quality is validated in RQ3.
% 语义标签质量在 RQ3 中验证。

Beyond label separation, we test whether choosing different field selections for the operation stack produces views that flat grouping cannot.
% 除标签分离外，我们还测试为 operation stack 选择不同字段是否产生扁平分组无法产生的视图。
We take the same 13{,}265 operations from the four oracle-rich families and fold them with five different field selections, counting how many distinct groups each produces.
% 我们取四个 oracle 丰富家族的同样 13,265 个 operation，用五种不同字段选择折叠，计算每种产生多少不同分组。
The same operations fold into 9~dataset groups, 57~phase groups, 226~semantic groups, 455~action groups, or 3{,}757~fixed-session groups by changing only the stack fields.
% 同样的 operation 仅通过改变栈字段即可折叠为 9 个数据集分组、57 个阶段分组、226 个语义分组、455 个操作分组或 3,757 个固定会话分组。
A flat GROUP BY on a single label cannot produce these views because different stack depths answer different questions (task-level budget vs.\ action-level duplication vs.\ session-level drilldown).
% 单标签的扁平 GROUP BY 无法产生这些视图，因为不同栈深度回答不同问题（任务级预算 vs. 操作级重复 vs. 会话级下钻）。
The same model also covers all 15 public trace families (Table~\ref{tab:datasets}) without type-specific objects.
% 同一模型也覆盖了所有 15 个公开 trace 家族（Table~\ref{tab:datasets}），无需特定类型对象。

Different views also produce different rankings.
% 不同视图也产生不同排名。
The \texttt{time} and \texttt{tokens} views of the same 325 sessions share only 7/10 top prompt categories (Spearman $\rho = 0.623$).
% 同一 325 个会话的 \texttt{time} 和 \texttt{tokens} 视图仅共享 7/10 的顶级提示词类别（Spearman $\rho = 0.623$）。
\texttt{network}-tagged prompts rank 8th by time (I/O wait) but 93rd by tokens, showing that the choice of weight function changes the analysis conclusions.
% \texttt{network} 标记的提示词按时间排第 8（I/O 等待）但按 token 排第 93，表明权重函数的选择改变了分析结论。

Finally, we test whether the profiler can produce useful groups without any user-specified fields.
% 最后，我们测试分析器能否在不需要任何用户指定字段的情况下产生有用的分组。
We run the automatic stack-induction mode (\S\ref{sec:stacks}) on the same six tasks used in RQ2 and compare the resulting group count and localization quality against flat summaries and fixed-session drilldown.
% 我们在 RQ2 使用的同样六个任务上运行自动栈诱导模式（\S\ref{sec:stacks}），将结果的分组数和定位质量与扁平摘要和固定会话下钻进行比较。
Induced stacks produce a median of 12 groups (versus 285 for fixed-session and 1 for flat), with 4/6 tasks generating variable-depth stacks.
% 诱导栈产生中位 12 个分组（固定会话 285 个，扁平 1 个），4/6 任务生成可变深度栈。
The induced stacks achieve lower AP than hand-configured stacks (median 0.276 vs 0.312) because they use only visible fields, but they reduce inspection work to 65.3\% versus flat summaries without requiring any user configuration.
% 诱导栈的 AP 低于手动配置栈（中位 0.276 vs 0.312），因为它们仅使用可见字段，但将检查工作量降至扁平摘要的 65.3\%，无需任何用户配置。

\subsection{RQ2: Does Profiler Output Correspond to Real Problems?}
% RQ2：分析器输出是否对应真实问题？

RQ2 tests whether the profiler's top-ranked groups actually contain real failures, safety violations, or wasted effort.
% RQ2 测试分析器排名靠前的分组是否确实包含真实的故障、安全违规或浪费的算力。
We use public benchmark datasets that provide ground-truth quality annotations (failure, safety violation, redundant step, human-task boundary), hide these annotations from the profiler, and check whether the top-ranked groups recover them, analogous to injecting known bottlenecks in a CPU profiler evaluation.
% 我们使用提供真实质量标注（故障、安全违规、冗余步骤、人工任务边界）的公开基准数据集，对分析器隐藏这些标注，检查排名靠前的分组是否恢复了它们，类似于在 CPU 分析器评估中注入已知瓶颈。

The benchmark uses six tasks across four datasets (AgentRewardBench looping and side-effect, SATraj-OS unsafe operation, AgentNet incorrect and redundant step, and OSWorld-Human group start), totaling 34{,}539 operations and 3{,}699 positives.
% 基准使用四个数据集的六个任务（AgentRewardBench 循环和副作用、SATraj-OS 不安全操作、AgentNet 不正确和冗余步骤、OSWorld-Human 组起始），共计 34,539 个 operation 和 3,699 个正例。

% 我们比较六种分组策略（扁平摘要、固定会话下钻、数据集原生层次、原始操作分组、operation stack 分析，以及使用隐藏标签的 oracle），按组包含的隐藏正例比例对组排序。

% 固定会话下钻快速找到首个正例（WTFP = 0.004），但将工作量碎片化为 285 个组，前五召回率仅 2.3\%。

% Operation stack 分析在前五组中检查 9.4\% 的工作量，在 30\% 检查预算内恢复 39.0\% 的正例，并将组数减少到 157.5。

% Figure~\ref{fig:rq2-tradeoff} 可视化了这一权衡：operation stack 分析位于低工作量、中等召回率区域。

% 该比较也作为 operation/operation stack 模型（\S\ref{sec:design}）的证据。
Fixed-session drilldown and dataset-native hierarchy are alternative abstractions that fix the hierarchy at session or dataset boundaries.
% 固定会话下钻和数据集原生层次是在会话或数据集边界固定层次的替代抽象。

% Operation stack 将两者作为特例包含（栈中包含 \texttt{session} 恢复固定会话；包含 \texttt{dataset} 恢复数据集原生）。

\subsection{RQ3: How Accurate Are the Labels?}
% RQ3：标签有多准确？

RQ1 and RQ2 assume that semantic labels (the labels derived by intent attribution or mapping rules) are useful.
% RQ1 和 RQ2 假设语义标签（由意图归因或映射规则派生的标签）是有用的。
RQ3 tests whether these semantic labels are accurate by comparing them against ground-truth annotations from the benchmark datasets.
% RQ3 通过将这些语义标签与基准数据集的真实标注进行比较来测试它们是否准确。
For free-form traces (real sessions), we measure whether the intent-attribution backends run reliably at scale and produce well-formed labels.
% 对于自由形式 trace（真实会话），我们测量意图归因后端是否在规模上可靠运行并产生格式良好的标签。
Semantic accuracy of the LLM/regex tagger on free-form prompts remains future work because none of the available datasets (67{,}304 rows across 6 sources) simultaneously expose free-form text and ground-truth semantic labels, making same-input accuracy evaluation infeasible with current data.
% LLM/正则标注器在自由形式提示词上的语义精度仍是未来工作，因为现有数据集（6 个来源共 67,304 行）没有同时暴露自由形式文本和真实语义标签，使得同输入精度评估在当前数据下不可行。

On 325 real sessions, the LLM tagger processes 118{,}021 tag requests with zero failures, 100\% syntax validity, and 70\% cache hit rate (operational details in RQ4).
% 在 325 个真实会话上，LLM 标注器处理了 118,021 个标注请求，零失败，100\% 语法有效性，70\% 缓存命中率（运行细节见 RQ4）。

Public benchmark datasets provide native action labels that serve as ground truth for mapping quality.
% 公开基准数据集提供原生操作标签作为映射质量的真实标准。
We run a leave-dataset-out evaluation.
% 我们运行留一数据集评估。
For each of 9 datasets, we train mapping rules on the other 8 and measure V-measure between the mapping-derived \texttt{phase} field and the dataset's native \texttt{action} labels, plus boundary F1 between mapped phase boundaries and native action boundaries.
% 对于 9 个数据集中的每一个，我们在其余 8 个上训练映射规则，测量映射衍生的 \texttt{phase} 字段与数据集原生 \texttt{action} 标签之间的 V-measure，以及映射阶段边界与原生操作边界之间的边界 F1。

\subsection{RQ4: What Is the Profiling Cost?}
% RQ4：分析开销是多少？

\sys is an offline, post-hoc profiler, so it adds zero overhead to the agent's runtime.
% \sys 是离线事后分析器，对智能体运行时不增加任何开销。
The profiling cost is the time and resources spent after the session ends.
% 分析开销是会话结束后花费的时间和资源。

We re-execute 76 tracked profiling configurations twice (152 invocations) to measure cost and verify determinism.
% 我们重新执行 76 个跟踪的 profiling configuration 两次（152 次调用）来测量开销并验证确定性。
All 76 specs produce byte-identical output across both runs.
% 全部 76 个 spec 在两次运行中产生字节相同的输出。
Median profiling time is 1.6\,s per spec (p95 2.8\,s, max 4.3\,s), and median output size is 37.5\,KB.
% 中位分析时间为每 spec 1.6 秒（p95 2.8 秒，最大 4.3 秒），中位输出大小为 37.5 KB。
The slowest specs process views with over 600 unique stacks, while the 60 robustness-check specs all complete within 1.6\,s.
% 最慢的 spec 处理超过 600 个唯一栈的视图，而 60 个稳健性检查 spec 全部在 1.6 秒内完成。
The full profile-spec composition path (operation files, query predicates, rank rules, and stack depth) stays deterministic across all 12 tested configuration variants.
% 完整的 profiling configuration 组合路径（operation 文件、查询谓词、排名规则和栈深度）在所有 12 个测试的配置变体中保持确定性。

Intent attribution adds a one-time cost for new prompts.
% 意图归因对新提示词增加一次性开销。
Of 118{,}021 tag requests on 325 real sessions, 70\% are cache hits (zero cost), and the remaining 35{,}136 new calls complete at p95 latency of 31\,ms per label using a local 3B-parameter model.
% 在 325 个真实会话的 118,021 个标注请求中，70\% 为缓存命中（零开销），其余 35,136 个新调用使用本地 3B 参数模型以 p95 延迟 31 毫秒/标签完成。
Grammar-constrained decoding ensures 100\% syntax validity (2{,}700/2{,}700 test labels).
% 语法约束解码确保 100\% 语法有效性（2,700/2,700 测试标签）。
Because labels are cached, re-profiling the same sessions with different stack fields or predicates incurs no additional LLM calls.
% 因为标签被缓存，用不同栈字段或谓词重新分析相同会话不产生额外 LLM 调用。

\paragraph{Limitations.}
% 局限性。
The real-session data comes from a single multi-month development project, so the label distributions may not generalize to other domains.
% 真实会话数据来自单个多月开发项目，标签分布可能无法推广到其他领域。
Semantic accuracy of the intent-attribution backends on free-form prompts remains unevaluated because no available dataset simultaneously exposes free-form text and ground-truth semantic labels (as discussed in RQ3 above).
% 意图归因后端在自由形式提示词上的语义精度仍未评估，因为没有可用数据集同时暴露自由形式文本和真实语义标签（见 RQ3）。
The evaluation uses fixed-session drilldown as a proxy for span-tree debugging tools, not a real OpenTelemetry or LangSmith trace import.
% 评估使用固定会话下钻作为 span 树调试工具的代理，而非真实的 OpenTelemetry 或 LangSmith trace 导入。

\section{Related Work}
% 相关工作

Flame graphs~\cite{flamegraphs} and pprof~\cite{pprof} establish the folded-stack lineage, where profiles are weighted samples with location hierarchies.
% 火焰图~\cite{flamegraphs}和 pprof~\cite{pprof} 建立了折叠栈的谱系，其中 profile 是带有位置层次的加权样本。
Pprof supports sample labels and can visualize them as pseudo stack frames through tag-root/tag-leaf options.
% Pprof 支持样本标签，并可通过 tag-root/tag-leaf 选项将其可视化为伪栈帧。
Perfetto generalizes trace ingestion with SQL-based analysis and derived events~\cite{perfetto}.
% Perfetto 通过基于 SQL 的分析和衍生事件~\cite{perfetto}泛化了 trace 摄取。
These systems support query-time aggregation, so \sys does not claim novelty for flexible aggregation alone.
% 这些系统支持查询时聚合，因此 \sys 不单独声称灵活聚合的新颖性。
The difference is structural: pprof's tagroot adds label-derived frames to an existing execution call stack.
% 差异是结构性的：pprof 的 tagroot 在已有执行调用栈上添加标签衍生帧。
Agent traces have no execution stack, so the entire profiling structure must be constructed from semantic fields.
% 智能体 trace 没有执行栈，因此整个分析结构必须从语义字段构建。
The evaluation also differs: \sys scores profiler output against hidden agent-trajectory labels, a methodology with no analogue in CPU profiling.
% 评估也不同：\sys 将分析器输出与隐藏的智能体轨迹标签进行评分，这一方法在 CPU 分析中没有对应物。

OpenTelemetry GenAI and OpenInference standardize GenAI spans~\cite{otel, openinference}.
% OpenTelemetry GenAI 和 OpenInference 标准化了 GenAI span~\cite{otel, openinference}。
LangSmith~\cite{langsmith}, Langfuse~\cite{langfuse}, and Phoenix~\cite{phoenix} provide trace visualization, session management, evaluations, and dashboards.
% LangSmith~\cite{langsmith}、Langfuse~\cite{langfuse} 和 Phoenix~\cite{phoenix} 提供 trace 可视化、会话管理、评估和仪表盘。
AgentOps~\cite{agentops} argues for structured agent observability covering goals, plans, tools, and artifacts.
% AgentOps~\cite{agentops} 主张涵盖目标、计划、工具和制品的结构化智能体可观测性。
AgentTelemetry~\cite{agenttelemetry} defines 14 fault types over 9 agent-specific span kinds that extend OpenTelemetry for agent orchestration phases.
% AgentTelemetry~\cite{agenttelemetry} 定义了 9 种智能体特定 span 类型上的 14 种故障类型，扩展了 OpenTelemetry 以覆盖智能体编排阶段。
Wang et al.~\cite{agenttraces-trust} survey evidence tracing and execution provenance as foundations for agent accountability.
% Wang et al.~\cite{agenttraces-trust} 综述了证据追踪和执行溯源作为智能体问责的基础。
These systems excel at single-execution debugging or fault detection.
% 这些系统擅长单次执行调试或故障检测。
\sys complements them with cross-session aggregate profiling.
% \sys 以跨会话聚合分析作为补充。
Datadog LLM Observability~\cite{datadog-llmobs} and Laminar Signals~\cite{laminar-signals} take steps toward semantic aggregation by clustering user messages or extracting structured events, but they characterize \emph{input distributions}, not \emph{resource attribution}.
% Datadog LLM Observability~\cite{datadog-llmobs} 和 Laminar Signals~\cite{laminar-signals} 通过聚类用户消息或提取结构化事件向语义聚合迈进，但它们刻画的是输入分布而非资源归因。

AgentRx~\cite{agentrx} releases annotated failure trajectories and uses constraint checking plus an LLM judge for root-cause localization.
% AgentRx~\cite{agentrx} 发布标注的失败轨迹，使用约束检查加 LLM 评判进行根因定位。
TELBench~\cite{telbench} builds span-localization benchmarks from semantic error annotations.
% TELBench~\cite{telbench} 从语义错误标注构建 span 定位基准。
AgentAtlas~\cite{agentatlas} argues that outcome leaderboards miss trajectory-level quality.
% AgentAtlas~\cite{agentatlas} 认为结果排行榜忽略了轨迹级质量。
TrajAD~\cite{trajad} and AgentFixer~\cite{agentfixer} detect trajectory anomalies and recommend fixes.
% TrajAD~\cite{trajad} 和 AgentFixer~\cite{agentfixer} 检测轨迹异常并推荐修复。
These systems perform \emph{span-level} or \emph{step-level} fault localization within a single execution.
% 这些系统在单次执行内执行 span 级或步骤级故障定位。
\sys performs \emph{category-level} aggregate profiling, identifying which categories of work produce the most failures, consume the most budget, or generate repeated system effects across sessions.
% \sys 执行类别级聚合分析，识别哪些工作类别产生最多故障、消耗最多预算或跨会话产生重复系统副作用。

\section{Conclusion}
% 结论

Agent observability needs profiling, not only debugging.
% 智能体可观测性需要性能分析，不仅仅是调试。
\sys shows that the semantic operation stack model, combined with pluggable intent attribution and stack construction algorithms, closes the aggregation gap for agent traces.
% \sys 表明，语义 operation stack 模型结合可插拔的意图归因和栈构造算法，填补了智能体 trace 的聚合缺口。
Scoring profiler output against hidden trajectory labels confirms that semantic profiling separates over 90\% of mixed system-effect weight, localizes labeled problems at 9.4\% inspection work versus flat summaries, and produces labels that agree with ground truth on 7/9 held-out datasets.
% 将分析器输出与隐藏轨迹标签评分确认，语义分析分离了超过 90\% 的混合系统副作用权重，以扁平摘要 9.4\% 的检查工作量定位标注问题，并在 7/9 保留数据集上产生与真实标签一致的标签。
Fixed-session drilldown remains faster to a first positive on 4/6 tasks, confirming that profilers and debuggers are complementary, and \sys supports both by including or excluding session fields in the stack query.
% 固定会话下钻在 4/6 任务上更快找到首个正例，确认分析器和调试器是互补的，\sys 通过在栈查询中包含或排除会话字段同时支持两者。
Multi-project evaluation and trace-ecosystem integration are the most impactful next steps.
% 多项目评估和 trace 生态系统集成是最有影响力的下一步。
